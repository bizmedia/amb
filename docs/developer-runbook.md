# Runbook разработчика (AMB)

Краткая пошаговая инструкция: локальный запуск, проект после регистрации, MCP и синхронизация агентов из registry.

## 1. Окружение и БД

- **Node 20**, **pnpm** (версия из корневого `package.json`).
- **PostgreSQL** и переменная **`DATABASE_URL`** для API (см. документацию по локальной БД).
- Применить миграции: `pnpm db:migrate` или `pnpm db:migrate:deploy` — в зависимости от сценария (разработка / CI).

## 2. Запуск приложения

```bash
pnpm install
pnpm dev
```

Поднимаются **web** (Dashboard / BFF) и **api** (Nest). Актуальные порты смотрите в выводе `pnpm dev` и в [README.md](../README.md) (часто web **3333**, API **3334**).

Проверка API: запрос к **`/api/health`** на базовом URL API.

**Docker из репозитория:** `docker compose up` — сервисы **postgres**, **api**, **web**. Отдельного контейнера **seed** нет.

## 3. Мультитенантность, учётная запись и проект

### Модель

- Система **мультитенантная**: данные изолированы по **tenant** (организация / рабочее пространство).
- У каждого тенанта **минимум один владелец** — пользователь, который создал тенант при первой **регистрации**. В API и JWT это роль **`tenant-admin`** (владелец тенанта).
- При **`signup`** в одной транзакции создаются **новый tenant** и **пользователь**, привязанный к этому tenant; отдельного «общего» тенанта для новых пользователей нет.
- **Post-MVP:** планируется возможность **пригласить коллег** в свой tenant (несколько пользователей на один tenant). **Сейчас** этого нет: у нового тенанта ровно **один** участник — владелец.

### Текущий поток после регистрации

1. При **регистрации** (`signup`) создаются **tenant** и **пользователь** (владелец); **проект не создаётся**.
2. После входа в **Dashboard** нужно **создать проект** вручную (проект принадлежит tenant из JWT).
3. Откройте проект и скопируйте **Project ID** (UUID) для MCP и интеграций.

**Замечание:** дефолтного пользователя в БД нет (отдельная миграция удаляет legacy `admin@local.test`). Старые миграции могли создавать строки с фиксированными UUID — в **текущем** приложении нет «особого» проекта по константе: проект создаётся только пользователем после входа. **Новые** пользователи получают **свой** tenant и видят только **свои** проекты (фильтр по `tenantId` из токена).

## 4. Опционально: bootstrap tenant (`AMB_BOOTSTRAP=true`)

При включённом **`AMB_BOOTSTRAP`** API только **upsert** default tenant (`Default Tenant` / slug `default`). **Пользователь не создаётся** — регистрация через signup в Dashboard. **Проект не создаётся** — его создают в UI.

## 5. MCP (Cursor и др.)

### 5.1 Зависимость

В проекте, из которого запускается MCP: пакет **`@openaisdk/amb-mcp`** (или локальный путь на `packages/mcp-server` в монорепозитории).

### 5.2 Переменные окружения

| Переменная | Назначение |
| ---------- | ---------- |
| **`MESSAGE_BUS_URL`** | Базовый URL **HTTP API** (тот же хост/порт, что у `/api/health`). Для опубликованного compose с отдельными портами UI и API — указывайте URL **API**, не только Dashboard. |
| **`MESSAGE_BUS_PROJECT_ID`** | UUID проекта из Dashboard. |
| **`MESSAGE_BUS_ACCESS_TOKEN`** или **`MESSAGE_BUS_TOKEN`** | JWT доступа к проекту, если API его требует (токен из Dashboard). |

Порядок чтения env в **`amb-mcp`**: переменные процесса → **`.cursor/mcp.env`** → **`.env.local`** → **`.env`**.

Пример шаблона: **`.cursor/mcp.env.example`** (скопировать в **`.cursor/mcp.env`**, не коммитить секреты).

### 5.3 Команда `amb-mcp setup`

Проверяет доступность API, наличие **`MESSAGE_BUS_PROJECT_ID`** и путь к registry. **Данные в шину не записывает** (ни агентов, ни тредов).

### 5.4 Первый вызов инструмента MCP

При **первом** вызове любого инструмента MCP (если задан **`MESSAGE_BUS_PROJECT_ID`**):

1. С диска загружается registry (см. ниже).
2. В API **досоздаются агенты** из registry, для которых в проекте ещё **нет** такой **роли** (`GET /api/agents` → при необходимости `POST /api/agents`).

**Registry:**

- Если есть **`registry.json`** (например `.cursor/agents/registry.json`) — используется он.
- Если **нет `registry.json`**, но есть каталог агентов с **`.md`** файлами (кроме `README.md`) — файл **`registry.json` генерируется автоматически**, каталог при необходимости создаётся.
- Если нет ни JSON, ни подходящих `.md` — синхронизация отработает с **пустым** списком; агенты в API не появятся, пока не появятся файлы или вы не заполните registry вручную.

Отладочный лог в stderr: **`AMB_MCP_BOOTSTRAP_LOG=1`**.

## 6. Треды и сообщения

Синхронизация при первом вызове MCP **только регистрирует агентов**. **Треды не создаются** автоматически — их заводите в **Dashboard**, через **API** или инструменты MCP (`create_thread` и т.д.).

## 7. Чего в этом репозитории больше нет

- Скриптов **`pnpm seed:agents`**, **`seed:threads`**, **`seed:all`**, **`seed:mcp-demo`** и контейнера **seed** в compose.
- Утилита **`pnpm seed:prd-message-task-linking`** — отдельный сценарий под PRD; к обычному onboarding не обязательна.

## 8. Итоговая цепочка

1. Миграции БД  
2. **`pnpm dev`** (или compose)  
3. Регистрация / вход  
4. **Создать проект** в Dashboard  
5. Скопировать **Project ID** (и при необходимости токен)  
6. Настроить **`.cursor/mcp.env`** / MCP в IDE  
7. **Первый вызов** MCP-инструмента — подтягивает агентов из **`.cursor/agents`** (`registry.json` или сгенерированный из `.md`)

## 1. По способу подключения


| Способ                                       | Кто использует                                               | Описание                                                                                                                                            |
| -------------------------------------------- | ------------------------------------------------------------ | --------------------------------------------------------------------------------------------------------------------------------------------------- |
| **REST API** (curl / fetch)                  | Скрипты, внешние сервисы, ручная отладка                     | Прямые HTTP-запросы к `/api/`*. Подходит для быстрых проверок и интеграций без SDK.                                                                 |
| **TypeScript SDK**                           | Свои приложения, воркеры, скрипты оркестрации                | Типизированный клиент `createClient(baseUrl)`: регистрация агентов, треды, отправка/получение, ACK, DLQ, polling inbox.                             |
| **MCP (Cursor / Codex / Claude Code и др.)** | ИИ-агенты в Cursor, Codex, Claude Code и других MCP-клиентах | Инструменты: `list_agents`, `create_thread`, `send_message`, `get_inbox`, `ack_message`, `get_dlq` и др. Агент в чате выполняет действия через MCP. |
| **Dashboard UI**                             | Разработчик, наблюдатель                                     | Визуальный мониторинг: список агентов, треды, сообщения в треде, inbox выбранного агента, DLQ, ручной retry.                                        |
| **AMB как Docker-сервис**                    | Другое приложение в той же среде                             | Запуск AMB через `docker compose up -d`; своё приложение ходит по HTTP на `http://localhost:3333`.                                                  |
| **Копирование SDK в проект**                 | Свой проект без зависимости от репо AMB                      | `cp -r lib/sdk your-project/lib/message-bus-sdk` и `createClient(url)` в коде проекта.                                                              |

## 2. По типу потока сообщений


| Сценарий                        | Описание                                                                                                          | Как реализовать                                                                                                                                                             |
| ------------------------------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Точка–точка (одному агенту)** | Сообщение одному получателю (`toAgentId` задан).                                                                  | `sendMessage({ ..., toAgentId: targetAgentId, payload })`. Inbox получателя видит только свои сообщения (+ broadcast).                                                      |
| **Broadcast (всем агентам)**    | Одно сообщение видит каждый агент в проекте.                                                                      | `sendMessage({ ..., toAgentId: null, payload })`. В inbox каждого агента попадают и прямые сообщения, и broadcast.                                                          |
| **Цепочка ответов (threading)** | Ответ на конкретное сообщение, связь родитель–потомок.                                                            | `sendMessage({ ..., parentId: incomingMessage.id, toAgentId: senderId, payload })`. В треде видна иерархия.                                                                 |
| **Последовательный workflow**   | Оркестратор создаёт тред и по шагам шлёт задачи агентам (PO → Architect → Dev → QA и т.д.).                       | Скрипт/оркестратор: `createThread` → цикл `sendMessage` по шагам, опционально ожидание ответа через polling. Пример: `pnpm orchestrator`.                                   |
| **Polling inbox**               | Агент или воркер периодически забирает входящие и обрабатывает.                                                   | `for await (const messages of client.pollInbox(agentId)) { ... }` или `getInbox(agentId)` в цикле с паузой. Настраиваемый интервал и AbortController для остановки.         |
| **Retry и DLQ**                 | Сообщения с истёкшим таймаутом доставки уходят в retry, после лимита попыток — в DLQ. Ручной или массовый повтор. | Retry: воркер `pnpm worker:retry` или API `POST /api/dlq/:id/retry`. Повтор всех: `POST /api/dlq/retry-all` или `client.retryAllDLQ()`. Просмотр: Dashboard или `getDLQ()`. |
| **Закрытие треда**              | Тред помечается как завершённый.                                                                                  | `PATCH /api/threads/:id` с `{ "status": "closed" }` или SDK `updateThread(id, { status: "closed" })`.                                                                       |

## 3. По роли пользователя


| Роль                            | Цель                                                                                            | Типичные действия                                                                                |
| ------------------------------- | ----------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------ |
| **Разработчик (локально)**      | Запустить шину, отладить агентов, проверить поток сообщений.                                    | `pnpm dev`, `pnpm seed:agents`, вызовы API/SDK или сценарии из `examples/`.                      |
| **Разработчик (интеграция)**    | Встроить AMB в свой сервис или скрипт.                                                          | Подключение по HTTP, использование SDK или MCP в своём коде/проекте.                             |
| **ИИ-агент в Cursor**           | Выполнять задачи пользователя через шину (создать тред, отправить задачу агенту dev/qa и т.д.). | Через MCP: вызовы `create_thread`, `send_message`, `get_inbox`, `ack_message` по запросу в чате. |
| **Оркестратор (скрипт/воркер)** | Автоматизировать сценарии: цепочка агентов, уведомления, пайплайны.                             | Скрипт на SDK: создание треда, рассылка по шагам, опционально ожидание ответов и закрытие треда. |
| **Наблюдатель / оператор**      | Следить за состоянием системы, тредов, DLQ.                                                     | Dashboard: просмотр агентов, тредов, inbox, DLQ; ручной retry при необходимости.                 |

## 4. Типовые сценарии по шагам

### 🔧 Регистрация агента и первое сообщение (SDK)

1. Запустить AMB и засеять агентов (или только зарегистрировать своего).
2. `createClient(url)` → `registerAgent({ name, role, capabilities? })`.
3. `createThread({ title })` → `sendMessage({ threadId, fromAgentId, toAgentId?, payload })`.
4. Получатель: `getInbox(agentId)` или `pollInbox(agentId)` → обработка → `ackMessage(msg.id)`.

### 🤖 Задача от пользователя к агенту через Cursor (MCP)

1. В Cursor настроен MCP для AMB (`MESSAGE_BUS_URL`).
2. Пользователь в чате: «Создай тред "fix-login" и отправь задачу агенту dev: починить таймаут логина».
3. ИИ через MCP: `create_thread` → `list_agents` (найти dev) → `send_message` с задачей.
4. Агент dev (если у него есть воркер или другой потребитель inbox) обрабатывает и при необходимости отвечает/ack.

### 📢 Объявление всем (broadcast)

1. Оркестратор или любой агент: `sendMessage({ threadId, fromAgentId, toAgentId: null, payload: { type: "announcement", text: "..." } })`.
2. Все агенты видят это сообщение в своём inbox (наряду с прямыми).

### 🔄 Workflow PO → Architect → Dev → QA

1. Скрипт оркестратора: получить агентов по ролям, создать тред.
2. Отправить задачу PO → (опционально дождаться ответа).
3. Отправить задачу Architect → …
4. Отправить задачу Dev → …
5. Отправить задачу QA → …
6. Закрыть тред. Пример: `scripts/orchestrator.ts`, `pnpm orchestrator`.

### 📥 Воркер агента (постоянный потребитель inbox)

1. Процесс/сервис с SDK: `pollInbox(agentId, { interval, signal })`.
2. В цикле: обработать сообщения (по `payload.type`), вызвать `ackMessage(id)`.
3. При падении сообщения уйдут в retry/DLQ; можно повторить через API или Dashboard.

### 🛠 Мониторинг и повтор из DLQ

1. Открыть Dashboard → раздел DLQ или `GET /api/dlq` / `client.getDLQ()`.
2. Повторить одно: `POST /api/dlq/:id/retry` или `client.retryDLQMessage(id)`.
3. Повторить все: `POST /api/dlq/retry-all` или `client.retryAllDLQ()`.

### 📦 AMB как сервис для другого приложения

1. В среде (Docker Compose и т.п.): `docker compose up -d` — поднимается AMB.
2. Своё приложение: HTTP на `http://localhost:3333` (или другой хост) или копия SDK с тем же URL.
3. Регистрация своих агентов, треды, сообщения — как в любом сценарии выше.

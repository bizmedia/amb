# PRD: Привязка сообщений шины к задачам по `tasksTouched`

**Версия:** 1.4  
**Дата:** 2026-03-30  
**Статус:** Draft  
**Автор:** Product Owner Agent

**Связанные документы:** [PRD: ключи задач, эпики, спринты](./PRD-issue-keys-epics-sprints.md) · [PRD: модуль задач](./PRD-tasks-module.md) · [PRD: связи между задачами](./PRD-task-links.md) · [MCP Message Bus — правила агентов](../.cursor/rules/mcp-message-bus.md) · [Архитектура](./architecture.md)

## 1. Product Goal

Сделать так, чтобы сообщения Agent Message Bus (AMB) с **непустым** списком **`tasksTouched`** в payload были **явно связаны** с соответствующими **задачами проекта** (`Task` с человекочитаемым `key`, например `AMB-0007`).

Ценность:

- из **карточки задачи** видеть, какие агенты и в каких тредах оставили отчёты, затрагивающие эту задачу;
- из **просмотра треда / сообщения** переходить к затронутым задачам без ручного поиска по ключу;
- снижать разрыв между **дискуссией в шине** и **бэклогом** в одном проекте;
- заложить основу для отчётов («активность по задаче», «последнее сообщение по ключу»).

---

## 2. Problem Statement

Сейчас `tasksTouched` — **соглашение в JSON payload** (типично `completion_report` у агентов), но:

- связь **не материализована** в данных: нет запроса «все сообщения, относящиеся к задаче `PPP-0042`» без полного сканирования payload;
- UI и API задач **не знают** о сообщениях шины как о первоклассной связи;
- дублируется смысл: ключи в тексте треда и в массиве не подкреплены целостностью на уровне БД/API.

---

## 3. Definitions

| Термин | Значение |
| ------ | -------- |
| **Сообщение** | Запись `Message` в AMB: `projectId`, `threadId`, `fromAgentId`, `payload`, … |
| **Задача** | Сущность `Task` в том же `projectId` с полем `key` (формат `PPP-0001`) |
| **`tasksTouched`** | Массив строк в `payload`; каждая строка трактуется как **ключ задачи** в рамках того же проекта, что и сообщение |
| **Триггер привязки** | Сообщение создано (или обновлено — см. §4) и `tasksTouched` — непустой массив после нормализации |

---

## 4. Scope

### In Scope

| Область | Описание |
| ------- | -------- |
| Разрешение ключей | Для каждого элемента `tasksTouched` найти `Task` с совпадающим `key` в **том же** `projectId`, что у сообщения |
| Персистентная связь | Хранить связь «сообщение ↔ задача(и)» так, чтобы её можно было выбирать индексами и без парсинга всего JSON payload при каждом запросе |
| Создание связей | На **создании** сообщения (основной путь: `send_message` / API создания сообщения) |
| Идемпотентность | Повторная доставка того же сообщения не должна плодить дубликаты связей |
| Невалидные ключи | Ключи без существующей задачи: зафиксированное поведение (см. §6) — не ломать создание сообщения |
| API | Чтение связей: по задаче — список сообщений (или ссылки на них); по сообщению — список затронутых задач (id/key/title минимум) |
| RLS / изоляция | Те же правила project-scoping, что у `Message` и `Task` |
| Dashboard | Минимально полезный UI: в деталях задачи — блок «Сообщения агентов» / ссылки в тред; в просмотре сообщения — чипы/ссылки на задачи |
| SDK / OpenAPI | Типы и эндпоинты (или поля вложения в существующие ответы) согласованы с API |
| MCP | Минимальная read-side возможность получить связанные данные через существующие task/message surfaces в контексте проекта |

### Out of Scope (v1)

| Исключение | Причина |
| ---------- | ------- |
| Автоматический **перевод статуса** задачи из сообщения | Отдельная политика PO + возможный follow-up |
| Редактирование `payload` с пересчётом связей | Сложность версионирования; при необходимости — отдельная story |
| Связь с задачами **другого** проекта / тенанта | Нарушение изоляции |
| Поддержка **UUID** внутри `tasksTouched` в v1 | Фокус на человекочитаемых ключах, как в конвенции агентов |
| Полнотекстовый поиск по всем payload на предмет упоминаний ключей | Неэквивалентно явному `tasksTouched` |
| Уведомления подписчикам задачи | Отдельная фича |

## 5. Product Decisions

| Вопрос | Решение (PO) |
| ------ | ------------- |
| Какие сообщения связывать | В v1 — **только** сообщения с `payload.type === "completion_report"` и непустым массивом строк `tasksTouched`. Любые другие типы payload в v1 не участвуют в materialized linking, даже если содержат поле `tasksTouched`. |
| Формат элементов массива | Строки в формате **task key** проекта (`PPP-0001`), без пробелов по краям; регистр как в БД (обычно верхний регистр префикса — согласовать с валидацией key). |
| Дубликаты в массиве | Одна связь сообщение↔задача **не более одного раза** даже при повторе ключа в массиве. |
| Несуществующий ключ | **Не блокировать** сохранение сообщения; ключ **пропустить** при создании связи; в v1 **не** возвращать `warnings.unknownTaskKeys` или эквивалентные API-visible предупреждения в API/SDK/MCP. Неизвестные ключи попадают только во внутренние логи/метрики. |
| Сообщение без `projectId` | Не применимо в текущей модели vNext — сообщение всегда в проекте. |
| Удаление задачи | Связи, где участвует удалённая задача, удаляются (**CASCADE** или эквивалент). |
| Удаление сообщения | Связи для этого сообщения удаляются. |
| Удаление треда | Поведение наследуется от политики удаления сообщений; связи очищаются вместе с сообщениями. |
| MCP scope v1 | В v1 **не** вводится отдельный универсальный write/read механизм для `tasksTouched`. Запись идёт через существующий `send_message` с `completion_report`; read-side MCP остаётся минимальным и backward-compatible: допускается только расширение существующих message/task surfaces в рамках отдельных stories, без нового общего top-level workflow для `tasksTouched`. |

*Способ хранения (junction-таблица vs массив UUID в JSON на Message) и индексы — решение **Architect** (ADR).*

## 6. Functional Requirements

### FR1. Извлечение `tasksTouched`

- При приёме сообщения сервер извлекает `payload.tasksTouched` **только если** `payload.type === "completion_report"`.
- Если `payload.type` отличается от `completion_report`, поле `tasksTouched` в v1 игнорируется для materialized linking.
- Если значение отсутствует, не массив или массив пустой — **ни одной** новой связи по этому правилу не создаётся.
- Если массив содержит элементы не-строкового типа, они **мягко игнорируются**; сообщение при этом не отклоняется и валидные строковые ключи продолжают обрабатываться.

### FR2. Разрешение в `Task`

- Для каждой валидной строки ключа выполняется поиск `Task` по `(projectId, key)`.
- Найденные задачи связываются с `messageId` согласно модели хранения.

### FR3. API: задача → сообщения

- Клиент с доступом к проекту может запросить список сообщений, **привязанных** к задаче (по `taskId` или по `key`).
- Ответ включает минимум: id сообщения, `threadId`, `createdAt`, `fromAgentId` (и при необходимости краткий preview payload — без обязательной тяжёлой выдачи всего JSON).

### FR4. API: сообщение → задачи

- В ответе **получения сообщения** или списка сообщений треда — опционально (query flag `includeLinkedTasks` или вложение по умолчанию) массив связанных задач: `id`, `key`, `title`, `state`.

### FR5. UI (Dashboard)

- На экране **деталей задачи**: отображается список связанных сообщений (с ссылкой на тред) или пустое состояние.
- В **просмотре сообщения/треда**: отображаются затронутые задачи как ссылки на детали задачи.

### FR6. Наблюдаемость

- Метрика или лог: число сообщений с непустым `tasksTouched`, число успешно разрешённых ключей, число пропущенных неизвестных ключей (без PII в логах).
- Неизвестные ключи наблюдаемы только через внутренние логи/метрики; пользовательские warnings в v1 отсутствуют.

### FR7. Минимальный MCP Scope

- В v1 агентам **не нужен** новый MCP-механизм для передачи `tasksTouched`: они продолжают вызывать существующий `send_message` и передавать `tasksTouched` только в `completion_report`.
- V1 не вводит отдельный универсальный top-level MCP tool специально для `tasksTouched`.
- Любая read-side поддержка в MCP должна быть минимальной и backward-compatible и реализуется как расширение существующих task/message surfaces в последующих stories этого эпика.

---

## 7. Non-Functional Requirements

- Запрос «сообщения по задаче» должен оставаться **приемлемым по latency** при росте числа сообщений (индексы по связям — в зоне Architect).
- Поведение при **конкурентном** создании сообщений — обычные гарантии БД (уникальность пары message–task).

---

## 8. Dependencies

- Наличие стабильного **`Task.key`** в проекте (E9A / существующая реализация).
- Согласованность конвенции агентов: `tasksTouched` содержит **те же** ключи, что в бэклоге проекта.

---

## 9. Acceptance Criteria (продуктовые)

1. Сообщение с `tasksTouched: ["PPP-0001"]` при существующей задаче с таким `key` в том же проекте создаёт **устойчивую** связь, видимую через API с обеих сторон.
2. Сообщение с несуществующим ключом **сохраняется**; связь для этого ключа не создаётся.
3. В UI деталей задачи пользователь видит хотя бы одно связанное сообщение для сценария из п.1.
4. Документация для агентов (правила MCP / internal) обновлена: при непустом `tasksTouched` связь с задачами **ожидается** на стороне сервера.

---

## 10. Final v1 Contract

### 10.1 Write Contract

- Сервер материализует связи message ↔ task **только** для сообщений, где `payload.type === "completion_report"`.
- Внутри `completion_report` сервер читает `tasksTouched` как массив task keys текущего проекта.
- Поддержка `tasksTouched` в произвольных payload в v1 **не вводится**.

### 10.2 Unknown Task Keys

- Если часть или все ключи из `tasksTouched` не существуют в текущем проекте, создание сообщения **не** завершается ошибкой.
- Неизвестные ключи **пропускаются** и не создают связей.
- API/SDK/MCP в v1 **не** возвращают `warnings.unknownTaskKeys` или аналогичные пользовательские предупреждения.
- Если наблюдаемость нужна, она реализуется только через серверные логи/метрики.

### 10.3 MCP Contract

- В v1 **не** добавляется новый универсальный MCP-механизм для `tasksTouched`.
- Запись полностью укладывается в существующий контракт `send_message`.
- Read-side расширения MCP допускаются только как минимальные, backward-compatible изменения существующих task/message surfaces в соответствующих implementation stories.

### 10.4 Out of Scope for v1

- `tasksTouched` в payload с любым `type`, кроме `completion_report`;
- API-visible warnings по неизвестным ключам;
- ручное управление или исправление materialized links через MCP/UI;
- backfill исторических сообщений;
- отдельный специализированный top-level MCP tool для `tasksTouched`.

---

## 11. Success Metrics (ориентиры)

- Доля `completion_report` с непустым `tasksTouched`, у которых **все** ключи разрешились в задачи (качество данных).
- Использование блока «сообщения по задаче» в UI (если доступна аналитика) или субъективная оценка в ретро.

### Декомпозиция в AMB (эпик, спринт, задачи)

#### Требование: блокеры и зависимости между задачами (как в Plane / Jira)

При **планировании работ** в AMB продукт должен поддерживать явные **зависимости между задачами** в духе [Plane](https://plane.so/) или Jira:

- связь **многие-ко-многим** между задачами одного проекта (`projectId`);
- у каждой связи — **тип (семантика)**, минимум:
  - **«Блокирует»** — направленная зависимость (источник блокирует цель; у цели видно «заблокировано»);
  - **«Связано с»** — общая связность без обязательной блокировки;
  - **«Дублирует»** — указание на дубликат / каноническую задачу (как в PRD по связям).
- полный перечень типов, правила направленности, запрет циклов для блокировок, API, RLS, UI и MCP — в **[PRD: Связи между задачами (Task Links)](./PRD-task-links.md)**; настоящий пункт фиксирует **продуктовое требование к планированию в AMB**, чтобы бэклог и доска отражали блокеры и связи, а не только текст в описании.

Скрипт читает `MESSAGE_BUS_*` из окружения, из `.cursor/mcp.env` или (legacy) из inline `env` в `.cursor/mcp.json`, создаёт эпик/спринт при отсутствии и заводит или **досвязывает** задачи по `title`. Инструменты MCP `create_task` / `update_task` поддерживают опциональные `epicId` и `sprintId` (после деплоя соответствующего API).

## 1. Навигация tasks-модуля: All Issues / Epics / Sprints

### Рекомендация

Использовать **горизонтальную навигацию в стиле вкладок с привязкой к маршрутам** (URL = источник истины):

- `/tasks` — All Issues (текущий список / board)
- `/tasks/epics` — список эпиков
- `/tasks/sprints` — список спринтов

Визуально — **shadcn `TabsList` + `Link`** (или набор `Link` с классами как у `TabsTrigger`) под строкой «Назад к дашборду» и заголовком, **до** переключателя List/Kanban (List/Kanban остаётся только на All Issues).

### Обоснование

- **Согласованность с дашбордом:** там первичное переключение контекста — вкладки (`Tabs`). Tasks-модуль получает тот же ментальный модель «режимов модуля».
- **Dropdown** для трёх равнозначных разделов ухудшает обнаруживаемость и не показывает «где я» без открытия меню.
- **Отдельный sidebar** внутри `/tasks` дублирует уже узкий layout и не используется на остальных экранах приложения — лишняя когнитивная нагрузка.
- **Маршруты** дают шаринг ссылок, кнопку «Назад» браузера и возможность позже добавить в Command Palette переходы на `/tasks/epics` и т.д.

### ASCII wireframe (шапка модуля)

```
┌──────────────────────────────────────────────────────────────────┐
│ [← Back to dashboard]                                            │
│ Tasks                                                            │
│ Project: Acme                                                    │
│                                                                  │
│ [ All Issues ] [ Epics ] [ Sprints ]    ← route tabs             │
│                                                                  │
│ (only on All Issues)  [ List | Kanban ]     [+ New issue]        │
└──────────────────────────────────────────────────────────────────┘
```

### Связь с эпиками

Детали эпика/спринта: `/tasks/epics/[id]`, `/tasks/sprints/[id]` — второй уровень; навигация «назад» ведёт к списку соответствующего раздела.

## 2. Issue key display (`AMB-0001`)

### Рекомендация

| Место | Представление |
|-------|----------------|
| **Таблица (list)** | Первая колонка или колонка сразу после key: моноширинный текст `font-mono text-xs`, опционально `Badge variant="outline"` если нужен визуальный якорь в плотной таблице |
| **Kanban-карточка** | Одна строка над title: key в `mono` + укороченный title; key **не** внутри огромного badge, чтобы не конкурировать с state/priority |
| **Заголовок диалога редактирования / future detail** | Слева или над title: `AMB-0001` в `mono`, вторичный цвет; title остаётся главным |

**Поведение:** key — **кликабельная ссылка** (копирование через контекстное меню браузера или отдельная иконка «Copy» в tooltip по желанию). Целевой маршрут: тот же экран задачи с фокусом (сейчас — диалог; при появлении `/tasks/issues/[key]` — прямой URL).

### Обоснование

- PRD требует узнаваемости ключа как у Jira/Linear — **mono** согласуется с [ux-design-system.md §2.3](./ux-design-system.md) (IBM Plex Mono для кода/идентификаторов).
- В списке ключ — **сканируемый идентификатор**; отдельная колонка упрощает сортировку визуально (даже если сортировка по key добавится позже).
- Badge на карточке оставить для **epic/sprint** (см. PRD FR5), чтобы не превратить карточку в «стену бейджей».

### ASCII (строка таблицы)

```
| AMB-0042  | Fix login redirect   | Backlog  | High  | … |
   ^mono       ^medium title
```

## 3. Epic assignment UX

### Рекомендация (MVP)

1. **Основной путь:** поля **Epic** в форме создания/редактирования issue — `Select` или **Combobox** (поиск по названию эпика при >~8 элементах), плюс пункт «Без эпика» / `null`.
2. **Быстрый просмотр:** в списке и на карточке — `Badge` с усечённым названием эпика (tooltip с полным названием); клик по badge опционально ведёт на `/tasks/epics/[id]`.
3. **Bulk:** не блокировать P1; заложить **тот же паттерн**, что в [ux-design-system §5](./ux-design-system.md): при появлении multi-select — панель «Выбрано N» и один dropdown «Назначить эпик».

### Обоснование

- В `tasks-module.tsx` уже везде **select** в формах и фильтрах — новое поле не должно вводить drag-and-drop как единственный способ (дороже в реализации и не описано в эпике как обязательное).
- Drag issue на «колонку эпика» на kanban конфликтует с текущим DnD по **state** — отдельная доска «по эпикам» = отдельный scope.
- Назначение с экрана эпика (добавить задачу в эпик) — полезная фаза 2: кнопка «Добавить задачу» с выбором из списка не в эпике или создание с презаполненным `epicId`.

## 4. Sprint lifecycle UX (PLANNED / ACTIVE / COMPLETED)

### Рекомендация

**Визуальное кодирование статуса**

| Статус | Badge (shadcn) | Дополнительно |
|--------|----------------|---------------|
| PLANNED | `variant="outline"` или `secondary`, нейтральный | Иконка календаря в списке |
| ACTIVE | `default` (primary) или обводка `ring-2 ring-primary/30` | В списке спринтов — **всегда первый** блок «Текущий спринт» при наличии ACTIVE |
| COMPLETED | `muted`, опционально `line-through` у дат | Не кликабельны действия Start/Complete |

**Кнопки на экране спринта (`/tasks/sprints/[id]`)**

- **Start sprint:** `Button` primary, видна только для `PLANNED`. Если API вернёт конфликт «уже есть ACTIVE» — кнопка disabled + `Tooltip` / inline alert с текстом из API.
- **Complete sprint:** `Button` **outline** или **secondary** (не destructive: завершение — нормальное действие), видна для `ACTIVE`. Подтверждение через **AlertDialog** с кратким текстом: «Незавершённые задачи останутся в проекте» (формулировка согласовать с PO).
- Для `COMPLETED` — только просмотр и список задач; назначение новых задач в спринт скрыто или disabled (согласно PRD).

### ASCII (список спринтов)

```
Sprints
────────────────────────────────────────
▸ CURRENT
  ┌────────────────────────────────────┐
  │ Sprint 12 · ACTIVE    [badge: live]│
  │ 3 Jan – 17 Jan    14 issues        │
  └────────────────────────────────────┘

  Planned
  • Sprint 13   PLANNED     [Start]
  ...

  Completed
  • Sprint 11   COMPLETED   (read-only)
```

### Обоснование

- Одновременно только один ACTIVE — пользователю нужен **один явный визуальный якорь** вверху списка.
- Complete не должен выглядеть как «удалить» — иначе ошибочные ожидания (см. design system §5 про destructive).

## 5. Фильтрация: epic + sprint + state / priority / assignee

### Рекомендация

- Расширить **существующий** блок фильтров в `Card` на All Issues: добавить **два `select`** (или Combobox): **Epic** и **Sprint** с опциями «Все», «Без эпика», «Без спринта» где уместно.
- **Логика:** все активные фильтры комбинируются по **AND** (как ожидается в большинстве таск-трекеров).
- **Быстрый пресет (опционально P1+):** чип или мини-кнопка «Только активный спринт» = установка `sprintId` активного спринта одним кликом (снижает трение при ежедневной работе).

### Обоснование

- Текущая сетка `md:grid-cols-6` уже плотная; при нехватке места — **вторая строка** той же `Card` или сворачиваемый блок «Ещё фильтры» с сохранением видимости state/priority/assignee в первой строке (самые частые).
- Разнос фильтров epic/sprint только на экраны Epics/Sprints не покрывает сценарий PRD «list / board support filter by sprint».

### ASCII

```
┌─────────────────────────────────────────────────────────────┐
│ State ▼  Priority ▼  Assignee ▼  Due from  Due to  Sort ▼   │
│ Epic ▼   Sprint ▼   [ Active sprint only ]                  │
└─────────────────────────────────────────────────────────────┘
```

## 6. Project settings — `taskPrefix`

### Рекомендация

- Разместить поле в **том же месте, где редактируется проект** сегодня: расширить **Manage project** dialog в `project-switcher.tsx` (секция «Task prefix» или «Идентификаторы задач») **или** выделить подпункт «Project settings» внутри этого диалога, если форма разрастаётся.
- **Label + Input** с подсказкой: «2–5 латинских заглавных букв; влияет только на **новые** задачи» (PRD: старые keys не меняются).
- **Валидация в реальном времени:** на `onChange` / `onBlur` — проверка формата (длина, uppercase, символы); ошибка **под полем** (`text-destructive text-sm`). Уникальность в tenant — **после submit** (сообщение API), плюс disable Save при локально невалидном значении.
- Показывать **read-only превью** следующего ключа: «Следующий номер: `AMB-0043`» если API отдаёт счётчик (опционально, улучшает доверие).

### Обоснование

- Отдельная страница настроек для одного поля раздувает навигацию; префикс — свойство **проекта**, не задачи.
- Разделение «локальная валидация» vs «сервер уникальности» снижает шум и соответствует типичному UX форм.

## 7. Empty states: Epics и Sprints

### Рекомендация

Следовать [ux-design-system §4](./ux-design-system.md): иконка приглушённая, заголовок, подпись, **один primary CTA**.

**Epics (`/tasks/epics`, список пуст)**

- Заголовок: например «Пока нет эпиков»
- Подпись: «Эпики помогают группировать задачи по инициативам»
- CTA: **«Создать эпик»** → открывает тот же Dialog, что и кнопка в шапке

**Sprints (`/tasks/sprints`, список пуст)**

- Заголовок: «Пока нет спринтов»
- Подпись: «Спринты задают итерации планирования для задач проекта»
- CTA: **«Создать спринт»**

**Empty после фильтра** (например, скрыты архивные): текст «Ничего не найдено» + **«Сбросить фильтры»** — как в design system.

### ASCII

```
        [ icon muted ]

      Нет эпиков

   Создайте первый эпик, чтобы
   сгруппировать задачи.

      [ Создать эпик ]
```

### Обоснование

- Консистентность с Agents/Threads empty на дашборде снижает стоимость поддержки и обучения.

## Дополнительные UX-заметки (вне прямых вопросов)

| Тема | Рекомендация |
|------|----------------|
| **Поиск по key** | Расширить существующий поток фильтров: отдельное поле «Key» или объединить с общим поиском с подсказкой «exact match»; при exact match одной задачи — опционально сразу открыть карточку (согласовать с PO). |
| **Epic → Sprint visibility** | На `/tasks/epics/[id]`: под заголовком эпика — компактная **легенда по спринтам** (мини-badges с именами спринтов и count) или группировка списка задач по спринту с подзаголовками «Backlog / Sprint 12 / …». |
| **Polling** | После Start/Complete спринта показывать короткий loading на кнопке; refetch списка спринтов; избегать полного мигания списка — skeleton только при первом заходе (как в §4 design system). |
| **Keyboard** | По аналогии с дашбордом: при появлении стабильных маршрутов добавить в Command Palette пункты «Tasks: All / Epics / Sprints» (§6 design system). |

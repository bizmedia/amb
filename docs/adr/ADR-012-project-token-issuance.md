# ADR-012: Механика выдачи project-токенов

Статус: Принято  
Дата: 2026-01-28

## Контекст

ADR-007 закрепляет JWT как механизм аутентификации с project-scoped токенами. Нужно определить конкретную механику:
- Выдача токенов (долгоживущие vs краткосрочные)
- Стратегия ротации
- Механизм отзыва (revocation)
- Аудит (createdBy, createdAt, lastUsedAt)

## Решение

Использовать **долгоживущие project-токены** с явной поддержкой ротации и отзыва.

### Модель выдачи

**Долгоживущие токены** (по умолчанию):
- Срок жизни: 90 дней (настраивается по тенанту)
- Выдаются tenant-admin или project-admin
- Хранятся в БД для учёта отзыва
- Подходят для интеграций machine-to-machine

**Краткосрочные токены** (улучшение в будущем):
- Срок жизни: 1–24 часа
- Выдача через endpoint обмена токенов
- Можно добавить позже без поломки существующих токенов

### Хранение и учёт

Каждый project-токен хранится в таблице `ProjectToken` (см. схему в полном документе): id, projectId, tenantId, name, tokenHash, issuedBy, createdAt, lastUsedAt, expiresAt, revokedAt.

### Поток выдачи

1. Админ запрашивает создание токена (через Dashboard или API): POST `/v1/admin/projects/:projectId/tokens`, тело `{ name, expiresIn? }`, нужна роль tenant-admin или project-admin.
2. Backend генерирует JWT с claims `{ sub: "project", tenantId, projectId, iat, exp }`, подписывает секретным ключом, сохраняет в БД только хеш (полный токен не хранить).
3. Полный токен возвращается один раз при создании; при последующих запросах — только метаданные (name, createdAt, lastUsedAt).

### Ротация

**Ручная** (MVP): админ создаёт новый токен; старый действителен до явного отзыва; возможна ротация без простоя.

**Автоматическая** (будущее): настраиваемый период ротации; новый токен выдаётся до истечения старого; переходный период.

### Отзыв

- Немедленный: PATCH по tokenId, установка revokedAt; при валидации JWT проверяется статус отзыва.
- Массовый: отзыв всех токенов проекта или тенанта (tenant-admin).
- Автоматический: при удалении проекта/тенанта или по истечении expiresAt.

### Валидация при запросе

Извлечь JWT из заголовка, проверить подпись и срок, по хешу проверить в ProjectToken (существует, не отозван, не истёк), обновить lastUsedAt, извлечь tenantId/projectId, установить контекст RLS.

### Аудит

Поля: createdBy, createdAt, lastUsedAt, revokedAt, expiresAt.

## Рассмотренные альтернативы

1) Только краткосрочные токены (в стиле OAuth): лучше безопасность, но сложнее для M2M — не для MVP.
2) API-ключи без JWT: проще, но JWT даёт стандартизацию (ADR-007).
3) Token exchange (краткосрочные из долгоживущих): возможное улучшение в будущем.

## Последствия

- Нужна таблица ProjectToken; валидация добавляет обращение к БД (можно кешировать).
- Отзыв в итоге консистентен (учёт TTL кеша).
- Нужны Admin API и UI Dashboard для управления токенами; SDK должен поддерживать хранение и использование токена.

## Заметки по реализации

- Кешировать результат валидации (Redis/память, TTL 5 мин); обновлять lastUsedAt батчами асинхронно.
- Не логировать полный токен (только хеш); rate limit на создание токенов; мониторинг злоупотреблений.

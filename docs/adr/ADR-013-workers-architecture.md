# ADR-013: Архитектура фоновых воркеров

Статус: Принято  
Дата: 2026-01-28

## Контекст

Системе нужны фоновые воркеры для:
- Повтора неуспешных сообщений (перевод из `delivered` в `pending` или `dlq`)
- Очистки старых сообщений (политика хранения)
- В перспективе: оркестрация, уведомления, аналитика

Нужно выбрать: общие воркеры на весь сервис или отдельные воркеры на тенанта/проект.

## Решение

Для MVP использовать **общие воркеры с учётом мультитенантности**.

### Архитектура воркеров

**Один процесс воркеров** (`apps/api/src/workers/`):
- Запускается как часть Nest.js приложения (или отдельным процессом)
- Обрабатывает все тенанты/проекты
- Использует контекст RLS для изоляции данных
- Допускает горизонтальное масштабирование (несколько инстансов)

### Типы воркеров

1. **Retry Worker**
   - Ищет сообщения в статусе `delivered` старше порога (по умолчанию 60 с)
   - Переводит в `pending` при retries < MAX_RETRIES
   - Переводит в `dlq` при retries >= MAX_RETRIES
   - Запуск каждую минуту

2. **Cleanup Worker**
   - Удаляет старые сообщения по политике хранения
   - Настраивается по тенанту/проекту (по умолчанию 90 дней)
   - Запуск раз в день

3. **Будущие воркеры** (не в MVP): уведомления, аналитика, обработка DLQ.

### Обработка с учётом мультитенантности

Воркеры обходят тенанты и проекты: запрос активных тенантов → для каждого тенанта активные проекты → для каждого проекта устанавливается контекст RLS и обрабатываются данные. RLS гарантирует, что воркер не получает доступ к данным другого тенанта/проекта.

### Масштабирование

**MVP:** воркеры в том же процессе, что и API; одно развёртывание, один пул соединений к БД.

**Будущее:** отдельные процессы воркеров; координация через блокировки в БД или очередь сообщений; при большом масштабе — шардирование по тенанту.

## Рассмотренные альтернативы

1) Воркеры по тенанту: лучше изоляция, но сложнее развёртывание — избыточно для MVP.
2) Отдельный сервис воркеров: независимое масштабирование, но дополнительная сложность — можно вынести позже.
3) Очередь сообщений (RabbitMQ/Kafka): надёжнее, но дополнительная инфраструктура — для MVP достаточно опроса БД.

## Последствия

- Воркеры должны соблюдать политики RLS; нужна роль БД с BYPASSRLS или корректный контекст RLS.
- Единая точка отказа (смягчается несколькими инстансами).
- Нагрузка на БД от опроса (допустимо для масштаба MVP).
- Удобно добавлять новые типы воркеров.

## Реализация

Интеграция с Nest.js: cron-задача раз в минуту; обход тенантов и проектов; в транзакции — SET LOCAL app.tenant_id, app.project_id и вызов логики обработки. Варианты развёртывания: один процесс (API + воркеры) в MVP; в будущем — отдельные процессы для API и воркеров. Мониторинг: метрики выполнения, время обработки по тенанту/проекту, алерты при сбоях, отображение статуса воркеров в Dashboard.

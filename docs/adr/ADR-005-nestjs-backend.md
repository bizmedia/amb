# ADR-005: Выделение backend в отдельный сервис на Nest.js

Статус: Принято  
Дата: 2026-01-28

## Контекст

Сейчас в репозитории UI и HTTP API реализованы в одном Next.js приложении. Продуктовая стратегия предполагает:

- Развёртывание как hosted-сервиса (не только локально).
- JWT-аутентификацию и авторизацию.
- Мультитенантную изоляцию данных (тенанты → проекты).
- Операционные задачи (воркеры, наблюдаемость, лимиты), для которых удобнее отдельный backend runtime.

При этом Dashboard должен остаться частью продукта, но без прямого доступа к базе данных.

## Решение

Создать отдельный backend-сервис на Nest.js (`apps/api`), который станет единственным источником истины для:

- HTTP API (версионирование, например `/v1/...`)
- Аутентификации (проверка JWT) и авторизации (RBAC)
- Ограничения доступа по проектам и работы с БД (включая передачу контекста RLS)
- Фоновых воркеров (retry, cleanup, оркестрация)

Сохранить Next.js Dashboard как отдельное приложение (`apps/web`), общающееся с `apps/api` только по HTTP.

## Рассмотренные альтернативы

1) Оставить API-маршруты Next.js в качестве backend

- Плюсы: меньше компонентов; меньший рефакторинг.
- Минусы: композиция auth/middleware ограничена; воркеры сложнее выносить; связка UI и API сохраняется; остаётся риск доступа к БД из UI.

2) Использовать минимальный HTTP runtime (Hono/Express/Fastify) вместо Nest.js

- Плюсы: более лёгкий фреймворк; меньше навязанных решений.
- Минусы: больше собственной связки для auth, DI, модулей, тестов; ниже стандартизация для продукта с несколькими командами.

## Последствия

- Потребуется рефакторинг: выделение доменной логики в переиспользуемые пакеты (`packages/core`, `packages/db`, `packages/shared`).
- Потребуются явный контракт API и стратегия версионирования.
- Обеспечивается чёткое разделение: UI как клиент; backend как независимо разворачиваемый сервис.

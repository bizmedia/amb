# ADR-017: Enterprise Rules and Policy Engine Contract

Статус: Предложено  
Дата: 2026-03-29

## Контекст

Для B2B SaaS платформы требуется единый способ задавать и исполнять правила безопасности, доставки, стоимости и эксплуатации. Сейчас часть правил закреплена неявно в сервисах, часть только в документах.

Нужен формальный policy contract с единым форматом решения и точками применения.

## Решение

1. Принять taxonomy enterprise-правил:
- Security
- Data Governance
- Delivery
- Cost
- Audit
- Ops

2. Принять единый контракт `PolicyDecision`:
- `deny` — операция блокируется.
- `allow` — операция выполняется.
- `warn` — операция выполняется, но фиксируется audit/alert.

3. Зафиксировать порядок приоритета решений:
- `deny` > `warn` > `allow`.
- Если сработало несколько правил, итоговое решение выбирается по максимальной строгости.

4. Зафиксировать enforcement points:
- API ingress
- message send
- inbox pull
- worker execution
- admin actions

5. Зафиксировать требования аудита policy:
- Любой `deny` обязателен к аудиту.
- Любой `warn` для cost/security обязателен к аудиту.

## Рассмотренные альтернативы

1. Разрозненные проверки в каждом модуле:
- Плюсы: быстро внедряется локально.
- Минусы: непредсказуемость, дублирование, трудный аудит.

2. Только allow/deny без warn:
- Плюсы: проще контракт.
- Минусы: нет мягкой деградации и управляемого governance для v1 soft budgets.

## Последствия

1. Архитектурно policy становится first-class слоем Control Plane.
2. Все критичные операции должны иметь policy hook.
3. Появляется единая база для будущего policy engine и compliance-отчётности.

## Acceptance gates

1. Документирован единый `PolicyDecision` контракт.
2. Зафиксированы enforcement points для всех критичных потоков.
3. Определён приоритет конфликтующих policy-решений.
4. Определены обязательные audit правила для `deny` и `warn`.

# ADR-009: Инфраструктура хостинга (Kubernetes + Podman)

Статус: Принято  
Дата: 2026-01-28

## Контекст

Продукт vNext предполагает hosted-развёртывание с поддержкой мультитенантности. Необходимо выбрать платформу контейнеризации и оркестрации.

## Решение

Использовать **Kubernetes** для оркестрации и **Podman** для контейнеризации.

### Контейнеризация: Podman
- Runtime для сборки и запуска образов
- Совместим с образами Docker
- Поддержка rootless-контейнеров
- Удобен для локальной разработки и CI/CD

### Оркестрация: Kubernetes
- Стандарт индустрии для оркестрации контейнеров
- Поддержка мультитенантных развёртываний
- Автомасштабирование
- Service discovery и балансировка нагрузки
- Управление ресурсами и изоляция

## Рассмотренные альтернативы

1) Только Docker Compose
- Плюсы: проще для single-instance
- Минусы: нет нативной мультитенантности, ограниченное масштабирование, ручное управление

2) Docker Swarm
- Плюсы: проще, чем Kubernetes
- Минусы: менее зрелая экосистема, меньше возможностей

3) Nomad
- Плюсы: простой, гибкий
- Минусы: меньшая экосистема, менее распространён

## Последствия

- Нужны манифесты Kubernetes (Deployments, Services, ConfigMaps, Secrets)
- Образы собираются через Podman/Docker
- Локальная разработка: Podman Compose или minikube/kind
- CI/CD должен собирать и пушить образы
- Helm charts могут упростить автоматизацию развёртывания

## Заметки по реализации

- Начать с простого развёртывания в Kubernetes
- ConfigMaps для конфигурации
- Secrets для чувствительных данных (JWT-ключи, учётные данные БД)
- Рассмотреть Helm charts для production

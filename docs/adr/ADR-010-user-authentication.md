# ADR-010: Аутентификация пользователей (собственная таблица users)

Статус: Принято  
Дата: 2026-01-28

## Контекст

Для Dashboard нужна аутентификация пользователей. Варианты:
1. Собственная таблица пользователей с паролями
2. Интеграция с внешним Identity Provider (IdP)

## Решение

Использовать **собственную таблицу пользователей** для аутентификации.

### Реализация
- Модель `User` в схеме Prisma
- Хеширование паролей (bcrypt/argon2)
- JWT для сессий
- Без OAuth-провайдеров в MVP (можно добавить позже)

## Рассмотренные альтернативы

1) Внешний IdP (Auth0, Okta, Keycloak)
- Плюсы: надёжная аутентификация, SSO, меньше своего кода
- Минусы: внешняя зависимость, стоимость, избыточная сложность для MVP
- **Решение:** не в MVP, можно добавить позже

2) OAuth-провайдеры (Google, GitHub и т.д.)
- Плюсы: удобство для пользователей, не нужно хранить пароли
- Минусы: дополнительная сложность, зависимость от провайдеров
- **Решение:** не в MVP, можно добавить позже

## Последствия

- Нужно реализовать:
  - Endpoints регистрации/входа
  - Хеширование и проверка паролей
  - Управление сессиями (JWT)
  - Восстановление пароля (в будущем)
- Безопасность:
  - Требования к сложности пароля
  - Rate limiting на вход
  - Блокировка после неудачных попыток
  - Токены сброса пароля с истечением срока

## Возможные улучшения

- Подключение OAuth-провайдеров (Google, GitHub)
- SSO с внешним IdP
- Многофакторная аутентификация (MFA)
- Аутентификация без пароля

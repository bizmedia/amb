# ADR-006: Модель Multi-Tenant (Tenant → Projects) с изоляцией данных по проекту

Статус: Принято  
Дата: 2026-01-28

## Контекст

Сервис должен поддерживать несколько тенантов, у каждого — несколько проектов. Данные шины сообщений (агенты, треды, сообщения, inbox/DLQ) должны быть изолированы так, чтобы:

- Администраторы тенанта управляли проектами внутри тенанта.
- Интеграции (внешние сервисы/клиенты) работали в контексте одного проекта.
- Доступ между тенантами был невозможен по проектированию (включая политики на уровне БД).

## Решение

Вводится двухуровневая иерархия:

- `Tenant` — верхняя граница безопасности.
- `Project` принадлежит `Tenant` и является операционной границей для шины сообщений.

Все сущности шины привязаны к проекту:

- `Agent` принадлежит `Project`
- `Thread` принадлежит `Project`
- `Message` принадлежит `Project`

Для простоты политик БД и производительности запросов допускается денормализация `tenantId` в таблицы, привязанные к проекту (в дополнение к `projectId`).

## Рассмотренные альтернативы

1) Только тенант (без проектов)

- Плюсы: проще модель.
- Минусы: нельзя разделить окружения или нагрузки; неудобно поддерживать токены по проектам; плохой DX для тенантов с несколькими проектами.

2) Только проекты (без тенантов)

- Плюсы: проще авторизация для project-токенов.
- Минусы: теряется семантика управления тенантами; нет верхней границы для админов; сложнее биллинг/владение/соответствие требованиям.

## Последствия

- API становятся привязанными к проекту; контекст проекта берётся из auth (JWT).
- Админ-операции требуют контекста тенанта и RBAC.
- Нужна миграция: существующие данные привязываются к тенанту/проекту по умолчанию.

## Актуализация: онбординг и владелец (2026-04)

- При **регистрации** (`signup`) создаются **новый** `Tenant` и первый `User` с ролью **`tenant-admin`** (владелец tenant). Это не путать с legacy default tenant/project из миграций.
- Дополнительные пользователи в одном tenant (**приглашения коллег**) — **вне текущей версии** продукта (post-MVP).
- Операционное описание: `docs/developer-runbook.md` (§ 3).

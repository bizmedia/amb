# ADR-007: JWT Auth с project-scoped токенами, выдаваемыми админами тенанта

Статус: Принято  
Дата: 2026-01-28

## Контекст

Требуется аутентификация с первого дня и понятная модель интеграции для нескольких внешних проектов. Условия:

- Аутентификация на основе JWT.
- Админы тенанта управляют проектами и выдают учётные данные для интеграций.
- Интеграции аутентифицируются на уровне проекта (учётные данные привязаны ровно к одному проекту).
- Dashboard — часть продукта и требует аутентификации пользователей (отдельно от интеграций по проектам).

## Решение

Использовать JWT как стандартный механизм аутентификации.

Поддерживаются два типа субъектов:

1) User JWT (для Dashboard/UI и админ API)
- Субъект: `user`
- В claims — принадлежность к тенанту и роли.

2) Project JWT (для интеграций machine-to-machine)
- Субъект: `project`
- JWT содержит один scope: `tenantId` + `projectId`.

Админы тенанта могут выдавать учётные данные для проектов. Детали выдачи (долгоживущий JWT или краткосрочный через exchange) — вопрос реализации, но должны поддерживаться:

- Ротация (несколько активных токенов на проект)
- Отзыв (revocation)
- Аудит (createdBy, createdAt, lastUsedAt)

## Рассмотренные альтернативы

1) Только API-ключи (без JWT)

- Плюсы: максимально просто.
- Минусы: неудобно для RBAC и сессий в UI; слабая стандартизация; сложнее эволюция.

2) Только OAuth

- Плюсы: хорошо подходит для аутентификации пользователей.
- Минусы: избыточно для интеграций по проектам; всё равно нужны project-scoped учётные данные.

## Последствия

- Все API endpoints должны проверять JWT и извлекать из него `tenantId`/`projectId`.
- Нужна админ-поверхность (API + Dashboard) для управления проектами и токенами.
- Улучшается безопасность и становится возможным мультитенантный хостинг.

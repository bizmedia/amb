# PRD: Связи между задачами (Task Links)

**Версия:** 1.0  
**Дата:** 2026-03-26  
**Статус:** Draft  
**Автор:** Product Owner Agent

**Связанные документы:** [PRD: модуль задач](./PRD-tasks-module.md) · [PRD: ключи, эпики, спринты](./PRD-issue-keys-epics-sprints.md)

---

## 1. Product Goal

Дать проекту **явные связи между задачами** (issue-to-issue links) внутри одного проекта: блокировки, дубликаты, общая связность. Связи — **источник правды** в API и БД, чтобы:

- в UI отображать блокеры и связанные задачи по **человекочитаемому ключу** (`PPP-0001`);
- агентам и интеграциям (в т.ч. **Agent Message Bus / MCP**) задавать и читать зависимости **структурированно**, а не только свободным текстом в тредах.

Цель: покрыть сценарии координации разработки и работы агентов без введения полноценной иерархии подзадач и без автоматизации workflow.

---

## 2. Problem Statement

Сейчас зависимости между задачами можно выражать только в описаниях или сообщениях. Это приводит к:

- невозможности фильтрации и отчётов («что блокирует эту задачу»);
- дублированию смысла между чатом агентов и бэклогом;
- ошибкам при ручном вводе ключей без проверки существования связи в системе.

Базовый PRD модуля задач относил relations к Post-MVP; нагрузка от **координации агентов** и роста бэклога обосновывает выделение фичи в отдельный релизный инкремент.

## 3. Scope

### In Scope

| Функция | Описание |
| ------- | -------- |
| Типы связей | Зафиксированный enum (см. §4): минимум **BLOCKS**, **RELATES_TO**, **DUPLICATES** |
| Область | Только задачи **одного и того же** `projectId` |
| CRUD связей | Создание, удаление, просмотр списка связей для задачи |
| API / SDK | Эндпоинты и типы для связей; опционально вложение в ответ GET task |
| RLS | Те же правила project-scoping, что у `Task` |
| UI (MVP) | В карточке/деталях задачи: список связей с `key`, названием, типом; действие «добавить / удалить связь» |
| MCP (рекомендуется) | Инструменты или расширение существующих: создать/удалить связь, получить связи по `taskId` или `key` |

### Out of Scope

| Исключение | Причина |
| ---------- | ------- |
| Связи между **разными проектами** или тенантами | Сложность RLS и продуктовых правил |
| **Иерархия подзадач** (parent/child) | Отдельная фича; иная модель данных |
| Автоматические правила workflow («нельзя DONE, пока блокер не DONE») | Отдельное решение PO + ADR |
| Графовый визуал, критический путь | Усложнение UI без обязательной ценности в v1 |
| Синхронизация с внешними системами (Jira import) | Интеграции позже |
| Комментарии к связи | Достаточно типа связи в v1 |

## 4. Product Decisions

| Вопрос | Решение |
| ------ | ------- |
| Семантика **BLOCKS** | Направленная связь: **источник → цель** означает «источник блокирует цель» (цель *blocked by* источник). В API хранить одну directed edge. |
| **RELATES_TO** | Симметричная семантика для пользователя; **способ хранения** (одна запись на неупорядоченную пару vs две) — решение Architect (ADR). |
| **DUPLICATES** | Направленная: «A дублирует B» (A считается дубликатом, канон — B); отображение в UI должно быть однозначным. |
| Ссылка на задачу в API | Поддержать идентификацию по **UUID** и по **key** там, где уже принято для task (как в E9A для get-by-key). |
| Дубликаты связей | Одна и та же пара и тип не должны создаваться дважды (уникальный constraint на уровне БД — см. ADR). |
| Циклы **BLOCKS** | Запрет циклов при создании/обновлении на уровне **application** (понятная ошибка 400). |
| Связь задачи с самой собой | Запрещена. |
| Удаление задачи | Связи, где задача участвует как `from` или `to`, удаляются (**CASCADE** или эквивалент). |

## 5. Functional Requirements

### FR1. Модель связи

- Связь имеет: `id`, `projectId`, `fromTaskId`, `toTaskId`, `type` (enum), `createdAt` (и при необходимости `createdBy` — если появится аудит актёра; иначе out of scope).
- `fromTaskId` и `toTaskId` принадлежат одному `projectId`; несовпадение — ошибка валидации.

### FR2. API

- `POST` создание связи: тело содержит `fromTaskId` или `fromKey`, `toTaskId` или `toKey`, `type`.
- `DELETE` удаление связи по `linkId` или по паре + тип (один способ выбрать в дизайне API — предпочтительно по `linkId`).
- `GET` список связей для задачи: query к существующему ресурсу task, например `GET /api/projects/:projectId/tasks/:idOrKey/links` — входящие и исходящие с полем направления и типом.
- Ответы list/create/delete проходят через те же правила авторизации, что и задачи проекта.

### FR3. SDK

- Типы `TaskLink`, enum `TaskLinkType`.
- Методы: `client.tasks.listLinks(projectId, idOrKey)`, `client.tasks.addLink(...)`, `client.tasks.removeLink(...)`.

### FR4. UI

- В деталях задачи: секция **«Связи»** с группировкой или колонками «Блокирует» / «Блокирует эту задачу» / «Связано» / «Дубликаты» (копирайт согласовать с i18n).
- Добавление связи: выбор второй задачи (поиск по **key** или названию), выбор типа.
- Удаление связи из той же секции.

### FR5. Agent Message Bus / MCP

- Документировать в правилах агентов: для фиксации зависимостей использовать **API связей**, а не только `send_message`.
- Расширить MCP-сервер message-bus (при наличии эндпоинтов): минимум **list links**, **add link**, **remove link** с `projectId` и идентификаторами задач (UUID или key — как в API).

### FR6. Наблюдаемость и ошибки

- Ошибки: цикл BLOCKS, самоссылка, задача из другого проекта, дубликат связи, несуществующий key — предсказуемые коды (400/404/409) и сообщения для i18n на фронте.

## 6. Acceptance Criteria

### Epic E9D (предлагаемое имя): Task Links

- [ ] Можно создать связь **BLOCKS** между двумя задачами одного проекта; в списке связей целевой задачи видно «blocked by».
- [ ] Можно создать **RELATES_TO** и **DUPLICATES** согласно §4; тип отображается в UI.
- [ ] Нельзя создать вторую идентичную связь (та же пара и тип).
- [ ] Нельзя создать цикл из **BLOCKS** (проверка на сервере).
- [ ] Нельзя связать задачу с самой собой.
- [ ] При удалении задачи связи с её участием исчезают.
- [ ] Запросы из другого проекта / без доступа не раскрывают связи (RLS / авторизация).
- [ ] SDK покрывает сценарии list/add/remove.
- [ ] MCP (если в scope релиза) позволяет агенту добавить и прочитать связи по проекту.

## 9. Risks

| Риск | Митигация |
| ---- | --------- |
| Разная интерпретация BLOCKS у пользователей | Подписи в UI + короткая подсказка в документации |
| Симметрия RELATES_TO и дубли в БД | ADR: один канонический способ хранения |
| Рост размера ответа GET task | Вынос `links` в отдельный endpoint или опциональный `?include=links` |

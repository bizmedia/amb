# PRD: Issue Keys, Epics и Sprints

**Версия:** 1.1  
**Дата:** 2026-03-29  
**Статус:** Draft  
**Автор:** Product Owner Agent

---

## 1. Product Goal

Добавить в модуль задач проектно-ориентированную систему идентификации и планирования:

- каждая задача получает человекочитаемый ключ в формате `PPP-0001`
- `PPP` задаётся как короткий префикс проекта в настройках проекта
- задачи можно объединять в эпики
- задачи и эпики можно планировать по спринтам

Цель: приблизить управление задачами к привычной модели Jira/Linear/Plane, не ломая текущий project-scoped MVP.

Уточнение от 2026-03-29: детальная sprint analytics вынесена в отдельный PRD: [PRD-sprint-analytics.md](./PRD-sprint-analytics.md).

---

## 2. Problem Statement

Сейчас задачи идентифицируются только внутренним UUID, что неудобно для:

- ссылок в обсуждениях, документации и сообщениях агентов
- сортировки и визуального распознавания задач внутри проекта
- планирования задач по крупным инициативам
- планирования объёма работ по временным итерациям

Без эпиков и спринтов tasks-модуль покрывает базовый CRUD, но не покрывает сценарии продуктовой разработки и релизного планирования.

## 3. Scope

### In Scope


| Функция                  | Описание                                                                    |
| ------------------------ | --------------------------------------------------------------------------- |
| Project task prefix      | У проекта есть настраиваемый короткий префикс, например `AMB`, `API`, `OPS` |
| Issue key                | Каждая задача получает неизменяемый ключ вида `PPP-0001`                    |
| Sequential numbering     | Нумерация идёт отдельно внутри каждого проекта                              |
| Epic entity              | Отдельная сущность эпика внутри проекта                                     |
| Issue → Epic link        | Обычная задача может принадлежать не более чем одному эпику                 |
| Sprint entity            | Отдельная сущность спринта внутри проекта                                   |
| Issue → Sprint link      | Задачу можно назначить в один активный или будущий спринт                   |
| Epic → Sprint visibility | В UI видно, какие задачи эпика распределены по спринтам                     |
| Project settings         | Префикс настраивается в настройках проекта                                  |
| List / board support     | Ключ, эпик и спринт видны в списке и карточках задач                        |
| Filtering                | Фильтры по epic и sprint                                                    |
| Sprint analytics         | Burndown, Burnup, Scope Change и Throughput для спринта                    |


### Out of Scope

- Кастомные workflow states per project
- Несколько префиксов внутри одного проекта
- Перенумерация существующих задач при смене префикса
- Иерархия эпик → под-эпик
- Несколько спринтов на одну задачу одновременно
- Velocity и story points
- Автоматическое планирование задач в спринт

## 4. Product Decisions


| Вопрос                      | Решение                                                                |
| --------------------------- | ---------------------------------------------------------------------- |
| Формат task key             | `PPP-0001`, где числовая часть фиксированной длины 4 с ведущими нулями |
| Уровень уникальности prefix | Уникален в рамках tenant                                               |
| Уровень уникальности номера | Последовательность уникальна в рамках project                          |
| Смена project prefix        | Разрешена только для будущих задач; существующие keys не меняются      |
| Обязательность key          | Да, для каждой новой issue                                             |
| Обязательность epic         | Нет                                                                    |
| Обязательность sprint       | Нет                                                                    |
| Что такое epic              | Отдельная планировочная сущность проекта, объединяющая набор задач     |
| Что такое sprint            | Ограниченная по времени итерация с названием, датами и статусом        |

## 5. Functional Requirements

### FR1. Project Task Prefix

- У проекта есть новое поле `taskPrefix`
- Поле обязательно для всех новых проектов
- Для существующих проектов требуется backfill
- Допустимый формат: 2-5 символов, только латинские заглавные буквы
- Значение уникально в пределах tenant
- Значение редактируется в настройках проекта только пользователем с правами управления проектом

### FR2. Issue Key Generation

- При создании issue система автоматически присваивает следующий порядковый номер проекта
- Ключ формируется как `{taskPrefix}-{sequencePadded}`
- Sequence начинается с `0001`
- Ключ сохраняется отдельно в issue и больше не меняется
- Удаление задачи не освобождает её номер повторно
- Импорт/миграция существующих задач должна назначить ключи всем historical issues

### FR3. Epic Management

- Пользователь может создавать, редактировать, архивировать и просматривать эпики проекта
- Эпик содержит минимум: title, description, status, createdAt, updatedAt
- Задача может быть привязана максимум к одному эпику
- В карточке задачи отображается текущий эпик
- В представлении эпика виден список входящих задач

### FR4. Sprint Management

- Пользователь может создавать, редактировать, запускать, завершать и архивировать спринты проекта
- Спринт содержит минимум: name, goal, startDate, endDate, status
- Статусы спринта: `PLANNED`, `ACTIVE`, `COMPLETED`
- В проекте одновременно может быть не более одного `ACTIVE` sprint
- Задача может быть назначена максимум в один спринт
- Список и board поддерживают фильтр по sprint

### FR5. Views and Navigation

- В списке задач видны: key, title, state, priority, assignee, epic, sprint, dueDate
- В board/card view показываются минимум key, epic badge, sprint badge
- Доступен поиск по exact key
- В навигации tasks-модуля доступны разделы: `All Issues`, `Epics`, `Sprints`

### FR6. API and Integrations

- API возвращает `key` для issue во всех read/write ответах
- API поддерживает фильтр по `epicId`, `sprintId`, `key`
- API поддерживает CRUD для epics и sprints
- SDK типы и схемы поддерживают новые сущности и поля

### FR7. Sprint Analytics

- Для каждого спринта доступны метрики: `Burndown`, `Burnup`, `Scope Change`, `Throughput`.
- Все метрики строятся по временным точкам внутри диапазона дат спринта.
- В UI графики реализуются через стандартные chart-компоненты (`shadcn/ui`, area chart где применимо).
- Фильтры аналитики минимум: `project`, `sprint`, `assignee` (optional), `priority` (optional).
- Данные аналитики доступны в API для дальнейшего использования в Dashboard.

## 6. Acceptance Criteria

### Epic A: Project Prefix and Issue Keys

- В настройках проекта можно задать и изменить `taskPrefix`
- Валидация не позволяет сохранить некорректный prefix
- При создании issue ей автоматически назначается key формата `PPP-0001`
- Две параллельные операции создания issue не приводят к одинаковому key
- У существующих issues после миграции заполнен key
- В UI key виден в списке, карточке и деталях задачи
- API и SDK возвращают key для issue

### Epic B: Epics

- Пользователь может создать эпик внутри проекта
- Пользователь может назначить issue в эпик и снять назначение
- Issue не может одновременно принадлежать двум эпикам
- Список задач можно фильтровать по эпику
- У эпика есть отдельное представление со списком его задач

### Epic C: Sprints

- Пользователь может создать спринт с названием и датами
- Пользователь может перевести спринт в `ACTIVE` и `COMPLETED`
- В одном проекте нельзя иметь более одного `ACTIVE` sprint
- Пользователь может назначить issue в sprint и убрать её из sprint
- Список и board можно отфильтровать по sprint
- В интерфейсе видно, какие задачи входят в активный sprint

### Epic D: Sprint Analytics

- На странице спринта отображается `Burndown`.
- На странице спринта отображается `Burnup` (done vs total scope).
- Видна динамика `Scope Change` по задачам спринта.
- Виден `Throughput` (closed issues per day/week) в рамках спринта.
- Графики корректно работают при изменении scope спринта в процессе выполнения.

## 9. Risks

- Смена `taskPrefix` после запуска проекта может создать путаницу в коммуникации, даже если старые keys остаются неизменными
- Без конкурентно-безопасной генерации возможны коллизии ключей
- Epic и sprint добавляют новые сущности проекта, значит объём работ больше, чем просто расширение `Issue`
- Sprint planning останется project-level feature без дополнительной role model, пока RBAC не будет расширен отдельно

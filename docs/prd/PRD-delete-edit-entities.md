# PRD: Удаление и редактирование сущностей

**Версия:** 1.0  
**Дата:** 2026-03-15  
**Статус:** Draft  
**Связанный эпик:** [E8: Удаление и редактирование сущностей](./backlog.md#epic-8-удаление-и-редактирование-сущностей)

## 1. Product Vision

Пользователь Dashboard и интеграции через API могут **удалять** треды (вместе с сообщениями), агентов и проекты, а также **редактировать** название проекта и данные агента (имя, роль). Все деструктивные действия требуют явного подтверждения; каскад удалений документирован и предсказуем.

---

## 2. Текущее состояние

|----------|--------------|-------------|--------------------|--------------------|
| **Thread** | ✅ `DELETE /api/threads/:id`, сообщения удаляются каскадом | ✅ Меню + диалог подтверждения | — | PATCH статуса (open/closed/archived) есть |
| **Agent** | ❌ Нет | ❌ Нет | ❌ Нет | ❌ Нет |
| **Project** | ❌ Нет | ❌ Нет | ✅ `PATCH /api/projects/:id` (name) | ✅ В ProjectSwitcher (редактирование названия) |

**Цель доработки:** довести до полноты удаление агента и проекта, редактирование агента; зафиксировать и при необходимости доработать уже существующие сценарии (тред, проект).

## 3. Scope

### В scope (MVP доработки)

| Функция | Описание |
|---------|----------|
| Удаление треда | Подтверждено: уже реализовано (API + UI). При удалении треда удаляются все сообщения треда. |
| Редактирование названия проекта | Подтверждено: уже реализовано (API + UI в ProjectSwitcher). |
| Удаление агента | API: `DELETE /api/agents/:id`. UI: пункт меню «Удалить» в списке агентов + диалог подтверждения. |
| Удаление проекта | API: `DELETE /api/projects/:id` с каскадным удалением связанных данных. UI: действие в управлении проектами + диалог с описанием последствий. При удалении текущего проекта — переключение на другой. |
| Редактирование агента | API: `PATCH /api/agents/:id` (name, role). UI: редактирование имени и роли в списке агентов (inline или диалог). Приоритет P2. |

### Вне scope (на текущий релиз)

- Удаление tenant (только проект и ниже).
- Bulk delete (множественное удаление тредов/агентов одним действием) — остаётся в UX Design System как будущее расширение.
- Мягкое удаление (soft delete) с возможностью восстановления.
- Редактирование slug проекта (только name).

## 4. Требования по сущностям

### 4.1 Тред

- **Удаление:** При удалении треда удаляются все сообщения этого треда (каскад). API уже реализует: сначала `message.deleteMany({ threadId })`, затем `thread.delete`. UI: меню по строке треда → «Удалить» → диалог с текстом о необратимости и удалении всех сообщений → кнопки «Отмена» / «Удалить». После удаления выбранного треда — снять выбор или выбрать следующий тред (см. [UX Design System, п. 5](../ux/ux-design-system.md#5-деструктивные-и-управляющие-действия-удаление-и-расширения)).
- **Редактирование:** Редактирование заголовка треда — вне scope данной доработки (при необходимости отдельная история).

### 4.2 Агент

- **Удаление:**
  - API: `DELETE /api/agents/:id` (project-scoped, с учётом JWT и ProjectGuard). При удалении агента: связи `Issue.assigneeId` — либо `SetNull`, либо запрет удаления при наличии назначенных задач (решение в реализации; рекомендуется SetNull для простоты).
  - UI: в списке агентов — DropdownMenu по строке; разделитель; пункт «Удалить» (destructive). Диалог подтверждения: заголовок «Удалить агента?», описание последствий (например, агент исчезнет из списка и из выборов assignee), кнопки «Отмена» / «Удалить».
- **Редактирование (P2):**
  - API: `PATCH /api/agents/:id` с телом `{ name?: string, role?: string }` (project-scoped).
  - UI: возможность изменить имя и роль агента (inline-редактирование или диалог). Валидация (непустое имя и т.д.) и i18n.

### 4.3 Проект

- **Редактирование названия:** Уже есть: `PATCH /api/projects/:id` с `{ name }`. В Dashboard — в переключателе проектов (ProjectSwitcher) возможность редактировать название. Требование: валидация и отображение ошибок API (toast или inline).
- **Удаление:**
  - API: `DELETE /api/projects/:id`. При удалении проекта каскадом удаляются (в допустимом порядке с учётом FK и RLS): project tokens, issues, messages (или через удаление тредов), threads, agents, затем project. Либо полагаться на каскад в БД (если в schema настроен `onDelete: Cascade` для зависимых моделей от Project).
  - Ограничение: запрет удаления «default» проекта (опционально; если есть константа default project id/slug).
  - UI: в экране управления проектами (например, в ProjectSwitcher или отдельная страница настроек проекта) действие «Удалить проект». Диалог подтверждения: заголовок «Удалить проект?», описание последствий (удалятся все агенты, треды, сообщения, токены и задачи проекта без возможности восстановления). Кнопки «Отмена» / «Удалить». Если удалён текущий выбранный проект — переключить контекст на другой проект или показать пустое состояние с призывом выбрать/создать проект.

## 5. User Stories и Acceptance Criteria

### US-1: Удаление треда (проверка/документирование)

- **Как** пользователь Dashboard  
- **я хочу** удалить тред и все его сообщения  
- **чтобы** очистить ненужные данные.

**Критерии приёмки:**  
- В списке тредов есть действие «Удалить» в меню по строке.  
- При выборе открывается диалог с подтверждением и текстом о необратимости и удалении сообщений.  
- После подтверждения тред и все его сообщения удаляются; список обновляется; при удалении выбранного треда выбор снимается или переходит на другой тред.

**Статус:** Реализовано (проверить соответствие UX Design System и i18n).

---

### US-2: Удаление агента

- **Как** пользователь Dashboard  
- **я хочу** удалить агента из проекта  
- **чтобы** убрать неиспользуемого или ошибочно заведённого агента.

**Критерии приёмки:**  
- API: `DELETE /api/agents/:id` возвращает 204 или 200 без тела; при отсутствии агента или неверном project — 404.  
- Связанные задачи (issues) с assignee = этот агент: assignee сбрасывается (SetNull) или удаление запрещено при наличии назначений (решение за командой).  
- Dashboard: в списке агентов у каждой строки меню с пунктом «Удалить» (после разделителя, destructive). Диалог подтверждения с заголовком и описанием последствий. После успешного удаления агент исчезает из списка.

---

### US-3: Редактирование названия проекта

- **Как** пользователь Dashboard  
- **я хочу** изменить название проекта  
- **чтобы** отразить реальное назначение проекта.

**Критерии приёмки:**  
- API: `PATCH /api/projects/:id` с `{ name: string }` обновляет название (уже есть).  
- Dashboard: в переключателе проектов можно войти в режим редактирования названия и сохранить; при ошибке API отображается сообщение (toast или inline).

**Статус:** Реализовано (проверить обработку ошибок и валидацию).

---

### US-4: Удаление проекта

- **Как** пользователь Dashboard  
- **я хочу** удалить проект  
- **чтобы** убрать проект со всеми данными (агенты, треды, сообщения, токены, задачи).

**Критерии приёмки:**  
- API: `DELETE /api/projects/:id` удаляет проект и каскадом (или явно в коде) все связанные данные в рамках проекта.  
- Опционально: нельзя удалить default project.  
- Dashboard: в управлении проектами есть действие «Удалить проект». Диалог с заголовком, описанием последствий (каскадное удаление), кнопки «Отмена» / «Удалить».  
- Если удалён текущий выбранный проект — автоматическое переключение на другой проект или пустое состояние с призывом выбрать/создать проект.

---

### US-5: Редактирование агента (P2)

- **Как** пользователь Dashboard  
- **я хочу** изменить имя и/или роль агента  
- **чтобы** исправить опечатку или обновить роль.

**Критерии приёмки:**  
- API: `PATCH /api/agents/:id` с телом `{ name?: string, role?: string }` (project-scoped).  
- Dashboard: в списке агентов возможность редактировать имя и роль (inline или диалог). Валидация; ошибки API отображаются. i18n для подписей и сообщений.

## 6. API Reference

### Существующие (без изменений по контракту)

```
DELETE /api/threads/:id     — удаление треда (и сообщений)
PATCH  /api/projects/:id    — обновление проекта (name)
```

### Новые / уточнённые

```
DELETE /api/agents/:id      — удаление агента (project из контекста JWT)
PATCH  /api/agents/:id      — обновление агента (name, role) [P2]
DELETE /api/projects/:id    — удаление проекта (каскад)
```

Формат ошибок и коды (404, 403, 400) — в соответствии с текущим API (например, `MessageBusError`, JSON body с `code`/`message`).

## 8. Зависимости и риски

| Зависимость | Описание |
|-------------|----------|
| JWT + ProjectGuard | Все новые endpoints в scope проекта из токена. |
| RLS | Удаление/обновление только в рамках своего tenant/project. |
| UX Design System | Единый паттерн удаления и подтверждения. |
| Default project | Решение: разрешать ли удаление default project; при запрете — явная проверка в API и сообщение в UI. |

**Риски:**  
- Каскад удаления проекта: порядок операций и возможные FK; при необходимости явная последовательность delete в сервисе вместо полного полагания на DB cascade.

## 1. Product Goal

Обеспечить в AMB **единую, запрашиваемую память** о:

1. **Зависимостях между задачами** (блокеры, связи, дубликаты) — как в Plane / Jira, а не только в тексте описания или в чате.  
2. **Статусе задач** — текущее состояние и при необходимости **история переходов**, чтобы человек, UI и ИИ опирались на одни и те же факты.  
3. **Эпиках** — не только список задач, но и **сводка здоровья инициативы** (прогресс, блокировки внутри эпика).  
4. **Спринтах** — **объём итерации**, завершение, переносы и **видимость рисков** (заблокированные задачи в спринте).  
5. **Связи с тредами** — уже существующая линия [message ↔ task](./PRD-message-task-linking.md) дополняет картину «что обсуждали по задаче», не заменяя зависимости между задачами.

**Ценность:** планирование и работа с ИИ (агенты, MCP, промпты с `key`) опираются на **структурированные данные** в API/БД: меньше рассинхрона между бэклогом, статусом и обсуждениями.

---

## 2. Problem Statement

Сейчас (частично реализовано в продукте):

- У **задачи** есть `state`, `epicId`, `sprintId`, `key`; у **эпика** и **спринта** — свои статусы.  
- **Связей задача↔задача** с типами «блокирует / связано / дубликат» в модели может ещё не быть — зависимости «живут» в голове и в сообщениях.  
- **История смены статуса** задачи обычно не хранится: система «помнит» только последнее значение.  
- **Эпик** не обязан автоматически отражать агрегат по дочерним задачам; **спринт** — не обязан явно фиксировать переносы задач и итог закрытия итерации.  
- Без **сводок** (блокеры в спринте, прогресс эпика) планирование в UI и в сценариях с ИИ остаётся трудоёмким.

Итог: для «полноценной памяти» недостаточно одних полей `Task`/`Epic`/`Sprint` — нужны **первая очередь связей между задачами**, **опционально журнал статусов**, **согласованные правила сводок и жизненного цикла спринта**, плюс **интеграция** с уже запланированными message–task links.

## 3. Связь с существующими PRD

| Тема | Где детализировано | Роль в настоящем PRD |
|------|--------------------|----------------------|
| Типы связей BLOCKS / RELATES_TO / DUPLICATES, API, UI, MCP | [PRD-task-links](./PRD-task-links.md) | **Входит в продуктовый контур**; настоящий документ задаёт приоритет поставки и склейку с эпиками/спринтами/статусом. |
| Ключи задач, CRUD эпиков и спринтов, фильтры, аналитика спринта | [PRD-issue-keys-epics-sprints](./PRD-issue-keys-epics-sprints.md) | **База**; настоящий PRD расширяет поведение «памяти», не дублируя ключевую генерацию. |
| Сообщения шины ↔ задачи по `tasksTouched` | [PRD-message-task-linking](./PRD-message-task-linking.md) | **Смежная память** (активность агентов); не заменяет task links. |

Противоречий между документами быть не должно: при конфликте **уточняющее решение** фиксируется в настоящем PRD или в ADR.

## 4. Scope

### 4.1 In Scope (релизный контур «Planning Memory»)

| Блок | Описание |
|------|----------|
| **A. Зависимости задач** | Реализация согласно [PRD-task-links](./PRD-task-links.md): модель, RLS, API, SDK, UI секции связей, MCP по возможности. |
| **B. Память статуса задачи (v1)** | Минимум: **текущий** `Task.state` остаётся источником правды; опционально в той же волне или follow-up: **журнал переходов** (`fromState`, `toState`, `at`, `actorType`/`actorId` при наличии модели актёра). |
| **C. Эпик: сводка** | API (и приоритетно UI) отдают для эпика: число задач по `state`, список задач с входящими **BLOCKS** (заблокированы внешними задачами), без обязательного авто-перевода `Epic.status` в v1 (см. §5). |
| **D. Спринт: сводка и завершение** | Для спринта: агрегаты по `state`, список/счётчик **заблокированных** задач в спринте; **явное действие** «завершить спринт» с политикой для незакрытых задач (перенос в другой спринт / снятие со спринта — выбор PO в §5). |
| **E. Склейка в UI** | На деталях задачи: статус + связи + эпик/спринт + (после реализации PRD message-task) связанные сообщения. |
| **F. MCP / агенты** | Документированные сценарии: читать связи и сводки по задаче/спринту/эпику; менять статус задачи через API; не полагаться только на текст треда для зависимостей. |

### 4.2 Out of Scope (v1 настоящего PRD)

| Исключение | Причина |
|------------|---------|
| Иерархия parent/child у задач | Отдельная фича |
| Обязательное правило «нельзя DONE, пока есть не-DONE блокер» | Отдельное решение PO + ADR; в [PRD-task-links](./PRD-task-links.md) вынесено из v1 |
| Импорт Jira / синхронизация с внешними PM | Интеграции позже |
| Граф зависимостей, критический путь | После стабильного API связей |
| Полная копия Sprint Analytics из E9, если ещё не внедрена | Настоящий PRD не блокирует [FR7](./PRD-issue-keys-epics-sprints.md); метрики можно наращивать параллельно |

## 5. Product Decisions

| Вопрос | Решение (PO) |
|--------|----------------|
| Источник правды для зависимостей | Только сущность **TaskLink** (или эквивалент по ADR), не парсинг описаний и не только `tasksTouched` в сообщениях. |
| Источник правды для текущего статуса | Поле **`Task.state`**; переходы только через API (включая MCP), чтобы аудит был возможен. |
| Журнал переходов статуса | **Рекомендуется** в той же программе релиза, что и task links, или сразу следующим инкрементом; без него «память» ограничена последним снимком. |
| Авто-статус эпика по задачам | **v1:** сводки обязательны, **автоматическая** смена `Epic.status` — опционально (флаг «предложить DONE», ручное подтверждение). |
| Завершение спринта | Обязательно **явное** действие пользователя; система предлагает шаги для незакрытых задач согласно выбранной политике (например: «снять со спринта» / «перенести в спринт X»). |
| Связь с сообщениями | [PRD-message-task-linking](./PRD-message-task-linking.md) остаётся ортогональным: сообщения не создают task links автоматически из текста без явного API. |

*Детали enum связей, циклы BLOCKS, симметрия RELATES_TO — по [PRD-task-links §4](./PRD-task-links.md) и ADR.*

## 6. Functional Requirements (сквозные)

### FR-A. Зависимости (делегирование)

- Выполнить **все FR из [PRD-task-links](./PRD-task-links.md)** (модель, API, SDK, UI, MCP, e2e).
- В ответах **списка задач** в спринте или эпике сервер **может** (опциональный `include`) добавлять флаги `blockedByCount` / `isBlocked` для UX (конкретизация — Architect).

### FR-B. История статуса задачи

- **B1.** При каждом успешном изменении `Task.state` создаётся запись журнала с `taskId`, `projectId`, `fromState`, `toState`, `createdAt`.  
- **B2.** `actorType` / `actorId` — если в системе есть стабильный актёр (user / agent); иначе допускается `system` / null с последующим уточнением.  
- **B3.** API: `GET .../tasks/:idOrKey/state-history` с пагинацией (новые сверху или снизу — единообразно с остальным API).  
- **B4.** UI: в деталях задачи collapsible «История статуса» (минимум список переходов и время).  

*Если журнал перенесён на фазу 2 — явно пометить в плане релиза; до этого FR-B считаются желательными, не блокирующими FR-A.*

### FR-C. Сводка эпика

- **C1.** `GET .../epics/:id` (или эквивалент) возвращает агрегаты: `taskCount`, `countByState` (map или массив).  
- **C2.** Опционально: `blockedTaskCount` — задачи эпика, у которых есть хотя бы одна входящая связь **BLOCKS** из задачи того же проекта.  
- **C3.** UI на странице эпика: отобразить эти сводки и список задач с индикацией «заблокирована» при наличии BLOCKS.

### FR-D. Сводка спринта и завершение

- **D1.** `GET .../sprints/:id` возвращает те же классы агрегатов, что и для эпика, по задачам с `sprintId = id`.  
- **D2.** UI спринта: сводка по состояниям + акцент на заблокированных задачах (ссылка на задачу и на блокер).  
- **D3.** Операция **«Завершить спринт»**: перевод спринта в `COMPLETED`; для каждой задачи спринта, не в `DONE`, применить выбранную политику (см. §5): массовое снятие `sprintId` и/или массовый перенос в выбранный спринт — **одна** политика на v1, задокументированная в API.  
- **D4.** Желательно: запись в журнале активности спринта или в метаданных о факте завершения и количестве перенесённых задач (упрощённый audit).

### FR-E. Согласованность документации агентов

- Обновить правила агентов (например `.cursor/rules/mcp-message-bus.md`, registry): зависимости фиксировать через **API task links**; статус задачи менять через **task state API**; в `completion_report` указывать `tasksTouched` для связи с [PRD-message-task-linking](./PRD-message-task-linking.md).

## 7. Non-Functional Requirements

- Запросы сводок эпика/спринта должны оставаться **приемлемыми по времени** при росте числа задач (индексы по `epicId`, `sprintId`, `state`; для блокировок — по графу связей — см. ADR).  
- Все новые сущности и запросы — под **RLS / project scoping**, как у `Task`.  
- Ошибки API предсказуемы для i18n на Dashboard.

---

## 8. Acceptance Criteria (уровень релиза)

1. Выполнены acceptance criteria [PRD-task-links §6](./PRD-task-links.md) для поставки связей.  
2. По эпику и по спринту в UI и API видны **агрегаты по state** и **индикация заблокированных** задач (после появления BLOCKS).  
3. Операция завершения спринта **не оставляет** незакрытые задачи в статусе «в завершённом спринте» без осознанного исхода (снято / перенесено) согласно выбранной политике.  
4. Если журнал статусов в релизе: виден хотя бы один переход после смены статуса задачи в UI.  
5. Документация для разработчиков и агентов отражает цепочку: **ссылки задач ↔ статус ↔ эпик/спринт ↔ сообщения (опционально)**.

## 11. Risks

| Риск | Митигация |
|------|-----------|
| Объём релиза слишком большой | Разбить: фаза 1 только task links + сводки без журнала; фаза 2 журнал + завершение спринта с политикой |
| Разные интерпретации «заблокирована» | Единая формула: входящий **BLOCKS** из задачи того же проекта |
| Перегруз GET task | Вынести links и history в отдельные endpoints или `?include=` |

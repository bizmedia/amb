# Участие в разработке

Спасибо за интерес к Agent Message Bus. Мы приветствуем контрибуции в виде исправлений ошибок, улучшений документации и новых идей.

## Как сообщить об ошибке или предложить фичу

- **Баги и ошибки:** создайте [Issue](https://github.com/bizmedia/amb/issues) с меткой `bug`. Опишите шаги воспроизведения, ожидаемое и фактическое поведение, версию Node.js и ОС.
- **Идеи и фичи:** создайте Issue с меткой `enhancement` или `feature`. Кратко опишите задачу и зачем это нужно.
- **Вопросы:** можно открыть Discussion или Issue с меткой `question`.

## Процесс контрибуции (Pull Request)

1. **Fork** репозитория и клонируйте свой fork локально.
2. Создайте **ветку** от `develop` (или от `main`): `git checkout -b fix/краткое-описание`.
3. Внесите изменения. Соблюдайте стиль кода (см. ниже).
4. Запустите **lint**: `pnpm lint`.
5. Убедитесь, что приложение собирается: `pnpm build`.
6. Сделайте **коммит** с понятным сообщением (например: `fix: обработка пустого inbox`, `docs: обновить QUICKSTART`).
7. Отправьте ветку в свой fork и откройте **Pull Request** в этот репозиторий.
8. Укажите в описании PR, что именно меняется и при необходимости привяжите к Issue (например: `Closes #123`).

## Требования к Pull Request

- Описание на русском или английском: что сделано и зачем.
- Связь с Issue — если PR закрывает задачу (в тексте: `Closes #номер`).
- Один PR — одна логическая цель (не смешивайте фичу и рефакторинг в одном PR без необходимости).
- Код должен проходить `pnpm lint` и `pnpm build`.

## Оформление Markdown (файлы .md)

При оформлении любых файлов Markdown используйте **продвинутые возможности Markdown и Emoji**, чтобы сделать текст максимально читабельным:

- **Структура:** заголовки (H1–H6), списки (маркированные и нумерованные), таблицы, цитаты (`>`), горизонтальные разделители (`---`).
- **Выделение:** **жирный**, *курсив*, `код`, ссылки с осмысленным текстом.
- **Расширенные возможности:** таблицы для сравнения и справочников, блоки кода с указанием языка, коллапсируемые блоки (`<details>`/`<summary>`), чеклисты (`- [ ]` / `- [x]`).
- **Emoji:** уместное использование эмодзи в заголовках и списках для быстрой навигации и визуальной структуры (например: 📋 для чек-листов, ⚠️ для предупреждений, ✅ для готового, 🔧 для настроек). Без перебора — в меру, чтобы не отвлекать от содержания.

Подробнее: [.cursor/rules/markdown.md](.cursor/rules/markdown.md).

## Стандарты кода

- **TypeScript:** проект на TypeScript; типы должны быть явными там, где это улучшает читаемость.
- **Lint:** используется ESLint (в т.ч. eslint-config-next). Перед отправкой PR выполните `pnpm lint` и исправьте замечания.
- **Стиль:** отступы и кавычки согласуются с существующим кодом в репозитории; при сомнениях ориентируйтесь на уже изменённые файлы в той же папке.

## Авторство

В коммитах будет виден ваш автор. При крупных изменениях можно добавить себя в список благодарностей в README или в комментарии к релизу — по договорённости с мейнтейнерами.

## Вопросы

Если что-то непонятно — откройте Issue с меткой `question` или обсудите в Discussions.
